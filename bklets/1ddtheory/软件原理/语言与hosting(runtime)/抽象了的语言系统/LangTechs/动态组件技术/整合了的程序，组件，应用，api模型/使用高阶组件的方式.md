组件非构件

构件是应用的构件，组件是程序的组件，也不是windows那种具有handle的可视控件类，这二者不必对应，形成程序发行即组件组合的效果。但如果能够这样做，自然更好(qt中组件即widget，widget可以形成有handle的win32 gui contrl)

组件对于一门语言来说是必须的，因为它首先是构件，而构件关乎运行环境，与runtime逻辑，是一门语言系统或显式或隐式都要解决的。

在《使用语言层的Interface dispatch-初阶组件》一文中我们谈到的组件主要是指程序的组成部件，就跟内存中的object tree一样，不过，一个组件往往对应内存中的一组对象，，，多个对象组成了一棵compent tree而已。

软件组合(由小到大的组合)，比软件创建容易(一体化的设计)，这方面，它跟class和object技术的目标是一样的。


构件，和接口编程，即对小粒度的oo对象，在内存中的那棵object tree进行interface dispatch化，使之更大的，易于管理的内存compent tree对象并进行复用。可以说是adv oo，oo技术的变形和增强


这也是OO的初衷，然而，它工作在抽象级，，源码的class级，而不是可视级和组件级。而这正是组件要解决的。oo应与面向二进制接口在技术上并列，前者s r c 级，后者构件级。

小到大，是粒度的自由组合，它的扩展完全在于各个小单位，而一体化的设计，，在于整个自身，不能就小粒度持续集成，复用


前面我们曾谈到小粒度oo对象的缺点，它们只工作在源码级，在高于源码级的构件级达到的复用性不够灵活和强大， 也即，组件不只是促成可视化，而是首先促成在内存中形成一棵compent tree的构件树。，，这样的好处是为某些类库形成一个逻辑单元，而这个逻辑单元可以作为发行单元，或应用单元。。。

相对传统可复用件的组合，组件编程还有其它更多的好处，大到框架设计模式，作为设计模式的产品级应用构，这一切，都可以用基于组件的可视化来代替。来简化。甚至它可代替rad和脚本编程。
